# Planview — Product Requirements Document

> **For coding agents (Cursor, Claude Code):** This document is the single source of truth for what Planview is, who it's for, and what it must do. Read this before writing any code. When in doubt about scope or behaviour, refer back here.

---

## What is Planview?

Planview is a **read-only desktop app** (Electron) that scans a developer's local filesystem for plan files generated by AI coding agents — specifically Cursor and Claude Code — and presents them in a unified, searchable interface.

The core problem it solves: AI agents create plan files as markdown during coding sessions. These files end up scattered across different locations depending on which tool created them and whether they were saved globally or per-project. A developer who wants to revisit a plan from two weeks ago has no good way to find it. Planview fixes that by being a single place to browse, search, and open all of those plans.

**This is a v1 utility app.** It does not edit plans, sync them, or manage them. It finds them and shows them.

---

## Who is it for?

Developers who use Cursor and/or Claude Code as their primary AI coding assistants and work across multiple projects. They generate plans regularly, often forget where they are, and want to reference past plans without digging through directories manually.

---

## Where agent plan files live

This is foundational knowledge for the scanner. Plan files are markdown (`.md`) files created in these locations:

### Cursor
| Scope | Path |
|---|---|
| Global (default) | `~/.cursor/plans/*.md` |
| Per-project (opt-in) | `<project-root>/.cursor/plans/*.md` |

### Claude Code
| Scope | Path |
|---|---|
| Global | `~/.claude/*.md` |
| Per-project | `<project-root>/.claude/*.md` |

The tool that created a plan is inferred from which directory path the file was found in — no file content analysis needed.

## How the app works — user flow

### First launch (onboarding)
1. App detects it's a first launch (no config exists)
2. Shows an onboarding screen explaining what Planview does
3. Asks the user to select one or more **project root directories** (e.g. `~/projects/`, `~/work/`) using a native folder picker
4. Saves selected roots to persistent config via `electron-store`
5. Immediately runs the first scan and shows results

### Every subsequent launch
1. App opens directly to the main plan list
2. Auto-scans in the background on launch and on every app focus (debounced 2s)
3. User can browse, search, and filter plans

### Viewing a plan
1. User clicks a plan card in the list
2. Detail panel opens on the right showing the full plan content
3. Checklist items are rendered as checked/unchecked (visual only, not interactive)
4. Buttons to open the file in the default editor or reveal it in Finder/Explorer

## Scanner behaviour

The scanner runs in Electron's **main process** only. It never runs in the renderer. Results are passed to the renderer via IPC.

### What it always scans (regardless of user config)
- `~/.cursor/plans/` — Cursor global plans
- `~/.claude/` — Claude Code global plans

### What it scans based on user-configured project roots
For each root directory the user added during onboarding, scan up to **3 levels deep** looking for:
- `<root>/<project>/.cursor/plans/*.md`
- `<root>/<project>/.claude/*.md`

### Directories to skip entirely
`node_modules`, `.git`, `dist`, `build`, `.next`, `out`, `coverage`, `.cache`

### Metadata extracted per plan file
| Field | Source |
|---|---|
| `title` | First `#` H1 heading in the file, fallback to filename without extension |
| `tool` | `"cursor"` or `"claude"` — inferred from directory path |
| `project` | Name of the parent project folder |
| `filePath` | Absolute path to the `.md` file |
| `lastModified` | `fs.stat().mtime` |
| `totalTasks` | Count of `- [ ]` and `- [x]` lines |
| `completedTasks` | Count of `- [x]` lines |
| `preview` | First 200 characters of body content (after the H1) |

### IPC contract
```ts
// Main exposes:
ipcMain.handle('scan-plans', async () => Plan[])
ipcMain.handle('get-config', async () => AppConfig)
ipcMain.handle('set-config', async (_, config: AppConfig) => void)
ipcMain.handle('open-file', async (_, filePath: string) => void)
ipcMain.handle('reveal-in-finder', async (_, filePath: string) => void)
ipcMain.handle('pick-directory', async () => string | null)
```

## UI layout

Three-column layout, fixed:

```
┌─────────────┬──────────────────────┬────────────────┐
│  Sidebar    │   Plan list          │  Detail panel  │
│  220px      │   flex: 1            │  280px         │
└─────────────┴──────────────────────┴────────────────┘
```

### Sidebar
- App name + icon at top
- Navigation: All plans, Recent (last 7 days), Completed
- Filter by tool: Cursor, Claude Code
- Filter by project: one entry per discovered project
- Plan count badge on each item

### Plan list (main area)
- Toolbar: title of current view, search input, filter button
- Scrollable list of plan cards
- Each card shows: title, tool badge, project name, relative time ("2 days ago"), 2-line content preview, progress bar (completedTasks / totalTasks)
- Clicking a card selects it and opens the detail panel
- Empty state when no plans found

### Detail panel
- Plan title and metadata (tool, project, last modified)
- Full checklist rendered — `- [x]` shown as checked, `- [ ]` as unchecked (read-only)
- Non-checklist markdown rendered normally
- Footer: "Open in editor" button, "Show in Finder" button
- Collapses / hides when nothing is selected

## Constraints and rules for agents

- **Never write filesystem operations in the renderer process.** All fs access goes in `electron/scanner.ts` and is called via IPC from the renderer.
- **Never make network requests.** This app is fully local. No analytics, no telemetry, no API calls.
- **v1 is read-only.** Do not implement any feature that modifies plan files.
- **Do not add features not listed here.** Scope creep will be handled in a separate PRD iteration.
- **shadcn/ui is the component library.** Do not install or use any other component library.
- **bun is the package manager.** Use `bun add` not `npm install`.
- **The app window is frameless** — implement custom drag region at the top of the sidebar/toolbar.
- **Follow the IPC contract exactly** as defined above. Do not add IPC channels not listed here without updating this document.

---

## Out of scope for v1

- Editing plan files from within the app
- Creating new plans
- Syncing plans to cloud
- Notifications or reminders
- GitHub / Git integration
- Any form of AI summarisation of plans
- Windows support (macOS only for v1)
